## Project Overview
Hider - 摸鱼阅读器，跨平台（macOS + Windows）桌面应用。
Electron + 原生 HTML/CSS/JS 架构，支持 TXT/MD/EPUB 文件阅读。

## Architecture
```
main.js              — Electron 主进程（窗口、托盘、全局快捷键、IPC、EPUB 解析）
preload.js           — contextBridge / ipcRenderer 桥接
renderer/
  index.html         — 主页面（设置面板、阅读区、标题栏）
  styles.css         — 全局样式（CSS 变量、暗色主题适配）
  app.js             — 渲染进程逻辑（设置控制、拖拽、快捷键、单行模式）
build/
  entitlements.mac.plist — macOS 签名权限
scripts/
  notarize.js        — macOS 公证脚本
```

## Run Commands
```bash
# 启动开发（注意：VSCode 终端需要 unset ELECTRON_RUN_AS_NODE）
npm start

# 构建 macOS
npm run build:mac

# 构建 Windows
npm run build:win
```

## Key Technical Notes
- VSCode 终端会设置 `ELECTRON_RUN_AS_NODE=1`，所有 npm scripts 已自动 `unset`
- 透明无边框窗口下 `-webkit-app-region: drag` 在 macOS 不生效，使用 IPC 手动拖拽实现
- 渲染进程沙箱化，不能直接访问 `process`，通过 `window.api.platform` 获取平台信息
- EPUB 解析在主进程使用 `adm-zip` 完成（ZIP → container.xml → OPF → spine → HTML → 纯文本）

## 核心交互原则（最高优先级）

### 理解优先原则（强制执行）
- 对于用户提出的问题或需求，**必须先不进行回答**
- 先对用户的话进行理解，然后用自己的语言复述回复，确认理解正确
- **等用户确认理解正确后，再进行实际的回答或执行**
- 禁止在未确认理解的情况下直接动手写代码或给出方案

### 第一性原理思维
- 拒绝经验主义和路径盲从，从根本目的出发思考问题
- 若用户动机模糊或目标不清晰，**立即停下讨论**，不要盲目执行
- 若当前路径非最优，**直接建议更短、更低成本的方案**，不要顺从低效路径

### 强制输出结构
所有回答必须分为两个部分：

**[直接执行]**
按照用户当前的要求和逻辑，直接给出任务结果。

**[深度交互]**
基于底层逻辑对用户的原始需求进行「审慎挑战」，包括但不限于：
- 质疑动机是否偏离目标（XY 问题检测）
- 分析当前路径的弊端与隐性成本
- 给出更优雅、更低成本的替代方案

## 变更自检规则（强制执行）

### 1. 修改前：调研验证
- 涉及**技术决策**时，先用 WebSearch 查询当前最佳实践
- 涉及**UI 决策**（布局、尺寸、间距、颜色）时，查询 Material Design / Apple HIG 等权威设计规范
- 涉及**安全决策**时，查询 OWASP 指南
- 不凭经验猜测，用调研数据支撑每一个决策
- 参考基准：8px 网格系统、桌面端面板宽度 280-360px、间距 8-24px

### 2. 修改后：自检清单
**前端 UI 自检：**
- [ ] 尺寸比例是否符合调研结论？与页面其他元素是否协调？
- [ ] 空间分布是否均匀？有无大面积留白或过度拥挤？
- [ ] 固定像素 vs 流式布局？不同窗口尺寸下是否合理？
- [ ] 同类组件的尺寸/间距/样式是否统一？
- [ ] CSS 变量是否在 :root 中集中管理？
- [ ] 暗色/亮色主题是否同步适配？

**代码自检：**
- [ ] 新增/修改的 IPC 通道是否在 preload.js 中注册？
- [ ] 是否存在内存泄漏（事件监听器未清理、定时器未清除）？
- [ ] 改动是否向后兼容？是否会影响现有功能？

### 3. 尽量一次做对
- 宁可多花 2 分钟调研，也不要让用户反复纠正同一个问题
- 如果不确定效果是否合适，主动告知决策依据并询问确认
- 不要在同一个问题上让用户纠正超过 1 次

### 4. 主动思考完整性（多想一步）
- 不要用户说什么就只做什么，要**主动推演关联影响**
- 修改某个组件时，必须检查：同类组件是否有相同问题？相关页面是否需要同步修改？
- 核心原则：**每次改动都要问自己「还有哪些地方会被这个改动影响？」**
- 如果推演出可能的关联影响但不确定是否需要改，在**[深度交互]**中主动反馈

## 前端 UI 设计规范

### 设计基准
- **网格系统**：8px 基础网格，所有间距为 8 的倍数（8/16/24/32px）
- **圆角**：小元素 6px，面板/卡片 8px，按钮 6-8px
- **字体层级**：标题 15px/600、正文 13px/500、辅助 11-12px/400
- **颜色系统**：通过 CSS 变量管理，亮色/暗色主题统一切换
- **过渡动画**：使用 `cubic-bezier(0.4, 0, 0.2, 1)` 缓动，时长 150-300ms

### macOS 原生风格
- 毛玻璃效果：`backdrop-filter: blur(24px) saturate(180%)`
- 微妙阴影：`box-shadow: 0 1px 2px rgba(0,0,0,0.04)` 用于浮起效果
- 交互反馈：hover 时 `background` 变化 + 轻微 `transform`
- 控件样式：toggle 开关、pill 形状标签、圆角输入框

### 暗色主题适配
- 使用 `.dark-theme` class 切换，通过 JS 亮度检测自动应用
- 暗色背景色：`rgba(28, 28, 35, 0.97)`
- 暗色边框：`rgba(255, 255, 255, 0.06-0.12)`
- 暗色文字层级：标题 #e0e0e0、正文 #aaa、辅助 rgba(255,255,255,0.25-0.35)

### 控件规范
- **滑块**：轨道 4px 高、圆形拇指 16px、hover 放大 1.15x + 阴影增强
- **颜色选择器**：32x32px、圆角 8px、hover 放大 1.05x
- **开关**：40x22px、圆形指示器 18px、cubic-bezier 过渡
- **按钮**：最小 padding 8px 14px、hover 上浮 1px + 阴影

## 下载与链接安全规则
- 依赖安装前确认包名正确（防 typosquatting）
- 不在代码中硬编码外部 CDN 链接
- `.gitignore` 包含 `.env`、密钥文件
- 文件导出使用 `Blob` + `URL.createObjectURL`，不引入外部下载服务
